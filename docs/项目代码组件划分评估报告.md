## 📊 项目代码组件划分评估报告

共扫描 **71 个** `.ts / .vue / .css` 源文件，其中 **16 个** 超过 300 行。整体来看，项目在中层模块（解析器、生成器、core）划分**比较合理**，但存在 **4 个明显需要优化的重灾区**。

---

### 🔴 P0 — 必须拆分

#### 1. `stores/editor.ts` — **3,795 行**

这是项目最大的问题。一个 Pinia Store 承载了超过 100 个函数，涵盖了 **State、Computed、Toast、项目打开/关闭、版本/迁移、SQL 导入、Undo/Redo、初始数据、DDL 生成、统一类型** 等全部职责。这是一个典型的「上帝对象」反模式。

内部虽有分段注释（`// ===== xxx =====`），但所有 action/getter 都平铺在一个 `defineStore` 闭包内，导致：
- 新人理解成本极高
- 任何修改都会 touch 这个 3000+ 行的文件
- 无法按需 lazy-load 不同功能模块

**建议拆分方案：**

| 子 Store | 职责 | 预估行数 |
|----------|------|---------|
| `stores/schemaStore.ts` | schemas/tables/fields CRUD、选中状态、展开状态 | ~600 行 |
| `stores/projectStore.ts` | 打开/关闭项目、加载/重载、文件句柄管理 | ~500 行 |
| `stores/baselineStore.ts` | 版本 CRUD、迁移 CRUD、diff 计算、DDL 预览 | ~600 行 |
| `stores/initialDataStore.ts` | 初始数据增删改、JSON/Table 模式切换 | ~400 行 |
| `stores/importSqlStore.ts` | SQL 导入解析、表名映射、导入目标选择 | ~300 行 |
| `stores/historyStore.ts` | Undo/Redo 命令管理、checkpoint 提交 | ~200 行 |

保留 `editor.ts` 作为组合入口（~200 行），内部组合以上子 store，对外保持 API 不变。

### 🟠 P1 — 强烈建议拆分

#### 2. `CommonConfigPanel.vue` — **1,242 行**

该组件内部包含两个完全独立的编辑表格：**统一类型管理** 和 **公共字段管理**。两者的拖拽逻辑、增删改逻辑各自独立，目前混在一个 SFC 中。

**建议：**
- 抽取 `components/panel/UnifiedTypesTable.vue`（统一类型表格 + 拖拽排序）
- 抽取 `components/panel/CommonFieldsTable.vue`（公共字段表格 + 拖拽排序）
- `CommonConfigPanel.vue` 降级为纯布局容器，只做 tab 切换

#### 3. `ProjectSettingsPanel.vue` — **1,053 行**

该组件承载了 4 个不同 tab 的全部模板和逻辑（全局配置、库结构、**版本管理**、项目设置），其中版本管理 tab 最为复杂（包含版本列表、迁移编辑器、DDL 预览面板）。

**建议：**
- 抽取 `components/panel/VersionManager.vue` — 版本管理 tab 的全部逻辑（~450 行）
- 抽取 `components/panel/VersionPreview.vue` — 版本快照预览面板（~200 行）
- `ProjectSettingsPanel.vue` 保留为 tab 容器

#### 4. `utils/file-helpers.ts` — **836 行**

混合了三层职责：旧结构读写 API + 新结构读写 API + 迁移逻辑 + UI 辅助函数。导出了 ~25 个函数，接口不够清晰。

**建议：**
- `utils/file-helpers/new-structure.ts` — 新结构（`current/` 目录）的读写 API
- `utils/file-helpers/old-structure.ts` — 旧结构读写 + 迁移逻辑
- `utils/file-helpers/display-utils.ts` — `parseDefaultInput`、`displayFieldLength` 等纯工具函数

### 📈 总结优先级

| 优先级 | 文件 | 行数 | 收益 |
|--------|------|------|------|
| 🔴 P0 | `stores/editor.ts` | 3,795 | 极高：降低认知负担，支持按需加载 |
| 🟠 P1 | `CommonConfigPanel.vue` | 1,242 | 高：每个子表格职责独立 |
| 🟠 P1 | `ProjectSettingsPanel.vue` | 1,053 | 高：版本管理是独立功能域 |
| 🟠 P1 | `file-helpers.ts` | 836 | 高：新旧结构 API 分离 |
| 🟡 P2 | `EditorSidebar.vue` | 532 | 中：拖拽逻辑复用 |
| 🟡 P2 | `InitialDataEditor.vue` | 677 | 中：编辑模式独立 |
| 🟡 P2 | `FieldTable.vue` | 684 | 中：展开行子组件 |
| 🟢 P3 | `create-table-parser.ts` | 1,171 | 低：高内聚，拆了反而不如 |

---

**总体评价：** 项目的模块划分思路是清晰的（`sql-parser/`、`sql-generator/`、`version/`、`structure-migrations/` 等目录拆分合理），最大的问题集中在 **Store 过大** 和 **几个面板组件承载了过多职责**。如果要优化，建议从 P0 + P1 的 4 个文件开始，能覆盖 **80% 的收益**。需要我针对其中某个制定详细的拆分方案吗？
