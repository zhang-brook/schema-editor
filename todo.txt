二、数据结构变更（大改动）
1. 基线概念
- 未来我们想做自动生成数据库变更SQL DDL自动维护，就是我们可以创建若干个基线版本，然后对于除了第一版外的每个基线版本，或者任选两个基线版本，都可以生成对应的变更脚本（这可能要对目录结构做一些调整，支持保存多个基线版本的数据。考虑到项目的工作目录可能会被用户使用git管理，我的初步设想是，当前基线使用固定文件或文件夹名称current，然后每创建一个基线，就生成一个对应的快照文件在本地）。关于生成变更sql，可能会有些问题，比如，field_a 变成 field_b，那么我们是采用rename还是删了再加（要么我们根据一个内部的唯一id来跟踪，要么让用户手动选择），用户还可以添加变更步骤，比如清空某列数据，对某列数据刷一个sql处理逻辑等等，还可以让用户自定义部分迁移sql，在这个项目中进行维护

2. initial-data 结构规范化
- initial-data 行内结构优化：目前 initial-data 的存储结构其实不是太合理（现状问题：rows / row_comments / field_comments / skip_rows 是 4 个平行数组，靠索引对齐，脆弱且难维护），或许我们可以优化一下，就是将 field_comments, skip_rows 直接跟记录行放在一起类似 rows: [ { data: 行的数据, field_comments: { field_1: 记录的注释, .... }, is_skip: 是否跳过 }]，这样能从根源上避免（但要引出一个新的问题之前，可以直接把array形式的数据 JSON  复制到文件中，那后面可能就要在页面上提供一个导入 JSON 数据的逻辑。）

3. (综上) 目录结构调整 规范化
- 目前的schema下所有表都保存在同一个sql中，或许未来我们可以将其拆分出来，每个表单独一个json，然后schema下只用一个数组引用及记录排序即可（类似common_config中记录的schema的名字及排序）
    目录布局预案（工作目录可能被 git 管理，所以基线用普通文件，以保留完整变更历史）：
    ```
    your-schema-folder/
    ├── common.json  # 记录与基线版本无关的配置
    ├── current/                    # 当前正在编辑的基线（固定名）
    │   ├── schemas/
    │   │   └── <schema_name>/ # 处理掉特殊字符后 文件名友好的模式名称
    │   │       └── <table_name>/ # 处理掉特殊字符后 文件名友好的表名称
    │   │           ├── table.json # 记录 table 配置项，包括字段定义等
    │   │           └── initial-data.json # 记录 table 配置项，包括字段定义等
    │   │       └── schema.json # 记录 schema 配置项，包含 schema原始名称 (也记录一下table排序)
    │   └── database.json # 记录 schema order 排序，以及所有与基线版本相关的之前在common_config中的配置项
    ├── baselines/                  # 历史基线快照 其中包含了 schema + table + initial-data (以及 schema + table 的 order)
    │   ├── 2026-07-09T16-32-v0.4.json
    │   └── 2026-08-01T10-00-v0.4.json                        # 【问】我想问下，baselines 是使用一个大json 合适，还是跟schemas结构保持一致合适？（我个人觉得和 schema 结构保持一致会更好，毕竟如果未来数据结构调整，需要对current和每个baselines版本都要做结构升级(也就是当前 version-upgrader.ts 做的事情)）
    └── migrations/                 # 用户维护的迁移脚本（自定义 SQL + 步骤）
        └── <migration-id>.json
    ```
    注意，目录变更时，最好考虑到能够未来目录变更的影响范围。最好在统一的位置去定义各个板块的目录文件在哪里或者目录拼接/获取方式
- 旧数据用升级器迁移，struct_version直接提到 '1.0'
- 目前还有一个问题，就是每步改动都要全量保存json和sql到本地甚至不止一次（不过未来我们调整后应该就可以解决这个问题）
  之后页面上的用户变更，只需写入对应json文件，而不要全量写入（生成出的sql最好也支持部分更新，但目前可先保持全量更新，这个问题暂时不大）

目前版本管理 - 创建基线这个功能设计有些不合理，现在是用的为每一个对象都创建一个唯一id来跟踪字段变化 field_id / table_id / schema_id / index_id / initial_data_id（行级） 创建
然后基线这边其实设计的配置逻辑也有些奇怪，如果不加以说明，会让用户难以上手。

我想的是，我们不再生成这些唯一id，然后允许用户创建多个基线版本，基线版本创建后，就可以按照创建时间形成一个类似时间轴的树状结构，然后每个迁移脚本，都关联一个起始脚本，和一个目标脚本
关联后，通过对比，可以知道删除了哪些表，新增了哪些表，未改名哪些表（比如原来是 a, b, c, d 三个表，用户将 b, c 名称对调，d 改为 e，我们得想个办法，让用户能够配置具体有哪些字段变化了，而且还要跟基线版本对齐。最好还能尽可能自动生成<这个地方我其实还有点不知道该怎么设计，想听听你的建议>）

我还想引入一个“环境”的概念，比如开发环境、生产环境等。每个环境都关联到一个基线上
此外，对于每个迁移都可以配置一些属性，比如该迁移在某个环境下是否已执行，迁移下的备注，迁移下对每个环境针对性的备注等

一个基线如果被迁移脚本、环境所关联，那么暂时不允许删除（可以通过tooltip或者其他形式告诉用户为什么 不让删除），如果没有被关联，则可以被删除
