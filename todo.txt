在这个类似"重构"的过程中，我们可以为这几点着重考虑：

一、样式板块
- 项目的UI太丑了，我们可以参考下GitHub上别的项目的优秀UI设计，来美化一下我们的项目，至少让用户用起来感觉很顺手，不要像在用几十年前的管理系统的体验

二、代码风格规范
- 保持当前代码风格不变

三、代码测试
- 是否接受引入 Vitest 作为测试框架（当前无测试依赖）？可以的。
- 引入 Vitest：对只测试逻辑即可，暂时先不接入 jsdom 的测试；@import 直接写进组件 <style scoped>
- 更新测试用例

四、不需要改动 或可以尽量保持现状的点
- 类型与实现混放 + 导出数据构建逻辑过于臃肿：我认为类型与实现混放 应该问题不大，或者说是我们的一个设计策略，如果把他们都抽出到全局types其实我们写的时候很容易忘记我们用的是哪个类型，如果我们只是模块内部的类型，就直接写在代码实现旁边，这样比较好找。当然，对于那些很多模块共用的类型，最好还是抽出到更高层级或者全局。
- Store 直接依赖 vue-i18n 其实问题倒不大，因为其中确实涉及到了i18n的信息，但如果你觉得要优化，确保最终界面功能要正常，不要因为测试把功能改的不能使用了

五、重构时务必注意
- [数据结构变更] 这几点是目前遇到的最头大的问题，我们的目标先不是直接上来解决，而是在当前重构优化的过程中，考虑到它们，
- git 分支提交：此外，重构的时候，请先在本地新创建一个 refactor/core-restructure 分支，然后按照每个提交最小化原子改动原则，将重构任务先拆分成若干原子改动，逐个改动后提交到本地，注意提交信息要符合团队协作规范，不要直接step1 xxx, step2 xxxx这样；修复加 fix: 功能加 feat:。并且每次提交前项目都要能够正常运行，不要提交一个编译报错的版本。
- 代码风格（重要！）：所有的重构，都请尽量保持我原有的代码风格，比如 .map.filter 的换行，注释，以及留空的行（我们一般在代码中对于一个逻辑语义放在一块，不同逻辑语义之间会留一个空行）。

六、碎碎念
- 其实目前功能改动牵涉到的地方还是很多的，未来最好能够减少模块间的耦合。比如，现在调整目录结构，包括但不限于会影响到：拖拽目录打开，本地磁盘文件变更检测，sql导入弹窗，schema和table的编辑，initial-data的处理，common_config的预设字段，保存文件和生成sql等等一大批位置
- 另外我比较担心的一点，可能是之前代码实现后其实从代码上看不出什么问题，但是界面上有时候会有些隐藏的小bug，所以在本本次重构的时候，如果涉及到函数的移动（比如从一个文件移到另一个文件，请确保完整移动而不是移动的同时重构，乳沟有重构需求的，可以在移动过去后，调整对应引用的import后git 提交一次，再进行重构，再提交一次，这样就可以追溯问题细节了；其他情况也是类似同理）

目前版本管理 - 创建版本这个功能设计有些不合理，现在是用的为每一个对象都创建一个唯一id来跟踪字段变化 field_id / table_id / schema_id / index_id / initial_data_id（行级） 创建
然后版本这边其实设计的配置逻辑也有些奇怪，如果不加以说明，会让用户难以上手。

我想的是，我们不再生成这些唯一id，然后允许用户创建多个版本版本，版本版本创建后，就可以按照创建时间形成一个类似时间轴的树状结构，然后每个迁移脚本，都关联一个起始脚本，和一个目标脚本
关联后，通过对比，可以知道删除了哪些表，新增了哪些表，未改名哪些表（比如原来是 a, b, c, d 三个表，用户将 b, c 名称对调，d 改为 e，我们得想个办法，让用户能够配置具体有哪些字段变化了，而且还要跟版本版本对齐。最好还能尽可能自动生成<这个地方我其实还有点不知道该怎么设计，想听听你的建议>）

我还想引入一个“环境”的概念，比如开发环境、生产环境等。每个环境都关联到一个版本上
此外，对于每个迁移都可以配置一些属性，比如该迁移在某个环境下是否已执行，迁移下的备注，迁移下对每个环境针对性的备注等

一个版本如果被迁移脚本、环境所关联，那么暂时不允许删除（可以通过tooltip或者其他形式告诉用户为什么 不让删除），如果没有被关联，则可以被删除

我正在做一款用json定义表结构，然后生成SQL脚本的工具。但是目前遇到了一些问题，就是我希望增加一个自动生成数据库版本变更后，自动生成变更脚本的逻辑。但是发现逻辑还有点复杂
如何实现，通过两个版本的建表语句差异，生成对应的数据库表结构变更SQL呢？我有点没有头绪。因为字段变更，不仅是删除和新增，还会有字段改名等情况，可能还要考虑到两列字段名互换等情况（更有甚者，会有多个字段循环改名的情况），感觉这个功能设计起来好复杂
此外，因为数据库迁移，除了像上面说的字段可以变，表、模式的名字也都可以变。除了表结构，表的数据也可以指定变更脚本。
我希望能够让使用该项目的用户，能够使用我这个schema-editor完全满足数据库结构设计和自动生成变更（同时保留手动修改）的需求

我觉得，这样是不是更合适：
因为我已经有版本（版本）的概念了，每个版本都是当时数据库结构那一瞬间的快照。或许，我在用户编辑表的时候，就记录一个 unversioned change set，把用户所有的变更都记录下来。然后用户生成迁移脚本时，就是直接获取这个 unversioned change set
但我又想了一下，或许这个不太合适，主要是用户可能还会有些其他的变更，以及对于同一个字段的多次修改不好合并。或许唯一id追溯是更合适的办法
